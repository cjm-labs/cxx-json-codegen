# CJM v0.2.0 Release Notes

CJM v0.2.0 is the Practical Models release.

This release turns the initial minimal generator into a layered compiler
pipeline with a practical C++ model subset, multi-header input support, and
stronger generated-code verification.

## Highlights

- Repositions CJM as a build-time metadata compiler
- Reorganizes the source tree into Core, C++ Frontend, and nlohmann Backend
  layers
- Extends the Metadata IR for practical field types
- Adds parser syntax facts for namespaces, enums, aliases, comments, and source
  locations
- Adds semantic analysis for v0.2 practical model mapping
- Resolves alias chains and detects alias cycles
- Resolves supported named types through current and parent namespaces
- Orders generated types by dependency
- Analyzes multiple explicit input headers together
- Generates practical nlohmann/json mappings for supported fields
- Supports repeated `--input` arguments in `cjm generate`
- Adds golden, CLI, semantic, and generated compile/run verification

## Supported v0.2 Model Surface

v0.2 supports a documented practical subset:

- `bool`
- signed integer types
- unsigned integer types
- floating-point types
- `std::string`
- enums
- nested generated structs
- namespaces
- type aliases within the supported syntax model
- `std::vector<T>`
- `std::optional<T>`
- field rename metadata
- ignore metadata through `json:"-"`
- `omitempty` for supported optional fields
- multiple explicit input headers

Example:

```cpp
namespace company::model {

struct Address {
    std::string city; // json:"city"
};

struct User {
    std::string name;                    // json:"name"
    int age = 0;                         // json:"age"
    std::vector<std::string> tags;       // json:"tags"
    Address address;                     // json:"address"
    std::optional<std::string> nickname; // json:"nickname,omitempty"
};

} // namespace company::model
```

## Multi-Header CLI Workflow

`cjm generate` now accepts repeated explicit input headers:

```bash
cjm generate \
  --input address.hpp \
  --input user.hpp \
  --output model.cjm.hpp
```

The CLI parses each explicit input, runs semantic analysis over the combined
source facts, and emits one deterministic generated header.

Automatic header discovery is still out of scope.

## Architecture

The repository now reflects the intended compiler layering:

```text
Source-language Frontend
        |
        v
Core Metadata IR
        |
        v
Code-generation Backend
```

Current implementation:

```text
C++ frontend
        |
        v
Metadata IR
        |
        v
nlohmann/json backend
```

The backend consumes Metadata IR only. It must not inspect C++ parser syntax.

```bash

cmake --build build-v02-verify
ctest --test-dir build-v02-verify --output-on-failure
```

Expected result:

```text
100% tests passed, 0 tests failed out of 15
```

The test suite includes:

- Metadata IR tests
- nlohmann backend golden tests
- parser tests
- semantic analysis tests
- CLI tests
- repeated multi-input CLI tests
- generated header compile/run tests
- basic example build integration tests

## Known Limitations

v0.2.0 intentionally remains a practical subset.

Not yet supported:

- `std::map<std::string, T>`
- `std::unordered_map<std::string, T>`
- arbitrary dynamic JSON values
- `std::variant`
- `std::any`
- pointer fields
- custom converters
- enum string mapping policies
- time and datetime mappings
- automatic header discovery
- full C++ grammar support
- production parser replacement
- install/package distribution
- cross-platform CI release gates

String-keyed maps and fixed-width numeric coverage are planned for v0.3.

Frontend parser research, including Tree-sitter evaluation, is planned for
v0.3.5.
