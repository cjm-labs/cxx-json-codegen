# Code Generator

The Code Generator is responsible for transforming the CJM Metadata Model into ordinary C++ source code.

It is the backend of the CJM pipeline.

The generator should not know how the source code was parsed.

It should not depend on parser-specific AST nodes.

It should only consume validated metadata.

---

# Overview

```text
C++ Source
    │
    ▼
Parser
    │
    ▼
Semantic Analysis
    │
    ▼
Metadata Model
    │
    ▼
Code Generator
    │
    ▼
Generated C++
```

The Code Generator is the final CJM-controlled stage before the user's normal C++ compiler takes over.

---

# Responsibilities

The Code Generator is responsible for:

- emitting generated C++ files
- generating serialization functions
- generating deserialization functions
- ordering generated declarations
- producing deterministic output
- preserving readability
- adding useful generated comments when appropriate

The Code Generator is not responsible for parsing or semantic validation.

---

# Input

The input to the Code Generator is the validated Metadata Model.

The generator may assume that:

- metadata is syntactically valid
- unsupported types have already been rejected
- duplicate JSON keys have already been detected
- type dependencies are known
- source locations are available when needed

If invalid metadata reaches the generator, that is a bug in an earlier stage.

---

# Output

The output is ordinary C++ source code.

Generated files use the naming convention:

```text
*.cjm.hpp
```

Example:

```text
user.hpp
```

may produce:

```text
user.cjm.hpp
```

Generated code should be valid, readable C++.

---

# Initial Backend

The first supported backend is:

```text
nlohmann/json
```

Initial generated functions may include:

```cpp
void to_json(nlohmann::json& j, const T& value);
void from_json(const nlohmann::json& j, T& value);
```

The exact generated shape may evolve as the implementation matures.

---

# Deterministic Output

Code generation must be deterministic.

The same input metadata should always produce the same output.

Deterministic output improves:

- debugging
- testing
- code review
- incremental builds
- reproducibility

Generated output should not depend on unordered container iteration or non-deterministic traversal.

---

# Readable Output

Generated code should be easy to inspect.

CJM should avoid producing unnecessarily clever or compressed output.

Readable generated code helps users:

- understand behavior
- debug problems
- trust the tool
- report issues

Generated code should resemble code that an experienced C++ developer might write manually.

---

# No Semantic Validation

The generator should not decide whether a field is supported.

It should not parse JSON tag syntax.

It should not resolve C++ types.

It should not build dependency graphs.

Those responsibilities belong to Semantic Analysis.

The generator should focus on one task:

```text
Metadata Model -> C++ source
```

---

# Dependency Ordering

Some generated types may depend on others.

Example:

```cpp
struct Address {
    std::string city; // json:"city"
};

struct User {
    Address address; // json:"address"
};
```

The Metadata Model should provide dependency information.

The generator uses that information to emit code in a valid order.

---

# Generated Comments

Generated files may include a short header comment:

```cpp
// This file was generated by CJM.
// Do not edit this file manually.
```

Generated comments should be concise.

They should not overwhelm the generated code.

---

# Error Handling

Most user-facing errors should be reported before generation.

If the generator fails, it should report an internal generation error with enough context to debug the issue.

Generator errors usually indicate:

- unsupported metadata model state
- invalid generation order
- file system failure
- implementation bug

---

# Testing

The generator should be tested using golden files.

A golden test compares generated output against an expected file.

Example:

```text
tests/golden/basic.input.hpp
tests/golden/basic.expected.cjm.hpp
```

Golden tests ensure that output remains stable over time.

---

# Future Backends

Although the first backend is `nlohmann/json`, the generator architecture should avoid unnecessary coupling to one backend.

Potential future JSON backends may include:

- RapidJSON
- simdjson-based helpers
- custom JSON adapters

However, the MVP should focus on one backend until the basic workflow is stable.

---

# Summary

The Code Generator is the backend of CJM.

It consumes validated metadata.

It emits ordinary C++.

It should be deterministic, readable, and easy to test.

The generator should never depend directly on parser-specific implementation details.
